## 2. Risk Areas

### 2.1 File-Based IPC Protocol (HIGH)

The entire communication protocol between the Pi agent process and the VS Code extension relies on filesystem operations in `.pi/review-requests/` and `.pi/review-results/`:
- **No atomicity** — partial writes can be read by the other process
- **No locking** — concurrent read/write races are possible
- **No transactions** — multi-step operations can be left in inconsistent states
- **No corruption recovery** — a crashed process leaves stale files

### 2.2 Silent Session Failures (MEDIUM)

`getCurrentSession()` in [vscode-ext/src/extension.ts:152](vscode-ext/src/extension.ts) silently fails when multiple review requests are pending. There is no fallback, error reporting, or queuing mechanism.

### 2.3 Synchronous File I/O Blocking (MEDIUM)

All file I/O uses synchronous APIs (`readFileSync`, `writeFileSync`, `mkdirSync`, etc.). In the VS Code extension host, this blocks the extension host process and can trigger VS Code's "extension is slow" warnings under load.

### 2.4 No Retry Logic (MEDIUM)

No file write operations have retry logic. Under filesystem contention (antivirus, network drives, WSL filesystem), transient failures will propagate as hard errors to the user.

## 4. Security Concerns

### 4.1 No Input Validation on IPC Messages (MEDIUM)

Review request file contents read in `handleRequest()` are parsed as JSON and used directly without any schema validation or sanitization. A malformed or malicious `.pi/` file could trigger unexpected behavior.

### 4.2 Path Traversal Surface (LOW)

The `resolveSafe()` function performs basic path normalization but does not validate that resolved paths stay within the workspace. Combined with the file-based IPC, this is a theoretical path traversal vector.

### 4.3 Trust Model (LOW)

The protocol trusts filesystem access control entirely. Any process with write access to `.pi/` can inject review requests. This is acceptable for a local dev tool but worth documenting.
