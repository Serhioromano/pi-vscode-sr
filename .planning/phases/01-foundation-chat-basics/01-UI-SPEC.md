# Phase 01 — UI Design Contract

> Visual and interaction contract for VS Code extension phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none |
| Icon library | VS Code built-in icons (`$(check)`, `$(close)`) |
| Font | VS Code default UI font (controlled by VS Code theme) |

**Rationale:** Phase 1 is a VS Code extension with no custom webview UI. All visual surfaces are VS Code native: Chat panel (`ChatResponseStream`), notifications (`showInformationMessage`/`showWarningMessage`/`showErrorMessage`), and the existing diff editor. No third-party UI framework, icon library, or custom styling is used. The extension integrates with VS Code's built-in UI surfaces and respects VS Code theme/accessibility settings.

---

## Spacing Scale

Not applicable. This phase has zero custom web UI. All rendering surfaces are VS Code native (Chat panel, notifications, diff editor) which use VS Code's own spacing and layout system. No pixel values are set by this extension.

---

## Typography

Not applicable. This phase has zero custom web UI. All text renders through VS Code's Chat panel (`stream.markdown()`, `stream.progress()`) and native notification APIs. VS Code controls all typography (font family, size, weight, line height, color) through its built-in theme and user settings. The extension passes plain text and markdown strings only.

---

## Color

Not applicable. This phase has zero custom web UI. No colors are set by this extension. VS Code's native surfaces (Chat panel, notifications) inherit the user's active theme and respect all accessibility contrast requirements.

## Copywriting Contract

This is the primary design deliverable for Phase 1. All user-facing copy is defined below.

### Chat Progress Messages (via `stream.progress()`)

| Event | Copy | Context |
|-------|------|---------|
| First message (lazy start) | `"Starting Pi..."` | `ChatRequestHandler` on first `@pi` invocation. Pi process spawns lazily per D-05. |
| `agent_start` | `"Pi is working..."` | Pi agent begins processing the request. |
| `turn_start` | `"Processing..."` | A new turn in the conversation starts. |
| `tool_execution_start` | `"Tool: {toolName}"` | Pi invokes a tool (write, edit, etc.). `{toolName}` is dynamically populated from the event. |
| `tool_execution_update` | (no copy) | Optional — used for tool progress updates if needed. Silent by default. |

### Chat Response Content (via `stream.markdown()`)

| Event | Copy | Notes |
|-------|------|-------|
| `message_update` (text_delta) | Dynamically streamed text content | Each `text_delta` fragment is passed to `stream.markdown()`. May buffer fragments to avoid markdown rendering artifacts across delta boundaries. |
| `agent_end` | None (no trailing marker) | Stream ends naturally. Phase 2 will add richer end-of-response signals. |

### Notification Messages

| Scenario | Surface | Copy | Action |
|----------|---------|------|--------|
| Pi not installed (D-07) | `showInformationMessage` (one-time) | `"Pi is not installed. Run: npm install -g @earendil-works/pi"` | Dismiss. Shown once per activation if `pi --version` check fails. |
| No workspace open | `showWarningMessage` | `"Pi Companion: open a workspace first"` | Dismiss. Extension returns early from `activate()` if no workspace folder. |

### Error Messages (shown in Chat panel)

| Scenario | Copy | Notes |
|----------|------|-------|
| Pi process fails to start | `"**Pi failed to start.** {error message}\n\nEnsure Pi is installed (\`pi --version\`) and try again."` | Shown in chat via `stream.markdown()`. User can retry by sending another message. |
| Pi process crashes unexpectedly (D-06) | `"**Pi process exited unexpectedly.**\n\n{stderr output}\n\nRun \`pi -c\` in terminal to resume the session. Send another message to restart Pi."` | Crash visible in chat for debugging. Notes explicit recovery path (`pi -c`). User controls restart by sending next message. No silent restart. |
| Generic execution error | `"**Something went wrong.**\n\n{error description}\n\nTry sending your message again."` | Catch-all for unexpected errors in `ChatRequestHandler`. |

### Interaction State: Lazy Start (D-05)

1. User types `@pi <message>` in VS Code Chat
2. Chat shows `"Starting Pi..."` via `stream.progress()`
3. `PiProcessManager.start()` spawns the Pi subprocess
4. On success: `"Pi is working..."` (from `agent_start`) then normal response flow
5. On failure: error message shown in chat, user can retry by sending again
6. Subsequent messages skip step 2-3 — instant response

### Interaction State: Crash Recovery Flow (D-06)

1. Pi subprocess exits unexpectedly while user is chatting
2. `pi-process-manager.ts` detects the crash via RpcClient exit/stderr
3. Chat stream receives crash error message (see above)
4. User sees the crash, reads stderr for debugging context
5. User runs `pi -c` in terminal to resume Pi session (optional)
6. User sends another `@pi` message — handler calls `PiProcessManager.start()` again
7. New Pi process spawns — session may be fresh (previous context lost — user informed via `pi -c` guidance)

### Interaction State: Pi Detection on Activation (D-07)

1. VS Code activates the extension (`onStartupFinished`)
2. Deferred init checks `pi --version` via `/bin/sh -c 'pi --version'`
3. **If found:** Continue with deferred init (heartbeat, watchers, pending review recovery)
4. **If not found:** Show one-time `showInformationMessage` with install instructions
5. Extension continues functioning for other features; Chat participant is registered but lazy start will fail with helpful error if user tries `@pi`

### Interaction State: Workspace Switch (D-08)

1. User switches VS Code workspace (active text editor changes to file in a different workspace root)
2. `extension.ts` detects the workspace root change
3. Current Pi session is persisted: call `client.newSession()` to ensure state is saved
4. Pi process for old workspace is stopped via `PiProcessManager.stop()`
5. Pi process for new workspace is created (lazy — no start yet) with new workspace root
6. If user returns to old workspace: saved session state is restored, Pi process re-created from persisted session file
7. No progress lost across workspace switches

### Primary Interactions (No CTA Buttons)

**No primary CTA button exists in this phase.** The primary user interaction is sending `@pi` messages in the VS Code Chat panel. The `@pi` participant is registered with `isSticky: true` so it remains selected across sessions. No destructive actions exist in this phase.
