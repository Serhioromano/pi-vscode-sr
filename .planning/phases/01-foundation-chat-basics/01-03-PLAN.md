And session state (sessionReviewIds, sessionApproveAll, projectCwd, vscodeNotOpenWarned) needs to be accessible from both. Since these are module-level variables, they need to either:
    a) Stay in src/index.ts and be passed to functions that need them
    b) Be in their own module (src/session-state.ts)
    c) Be in src/review-lifecycle.ts (since createReviewAndWait uses them)

    Looking at the current code:
    - sessionReviewIds is used in createReviewAndWait (line 51, 120-122) and session_start handler (line 425)
    - sessionApproveAll is used in createReviewAndWait (lines 53-56, 120-122) and before_agent_start handler (lines 443-445)

    Since D-02 says "Factory functions with closure-based state — no module-level mutable let/const state" but this applies to vscode-ext modules. The Pi extension is a different package. The Pi extension's module-level state is acceptable — it's a CLI process, not a VS Code extension.

    However, the module-level variables need to be accessible across the entry point and the extracted functions. The cleanest approach for the Pi extension:
    - Keep session state in src/index.ts as module-level variables
    - Export getter/setter functions or pass state explicitly to createReviewAndWait

    Actually, the simplest refactoring that doesn't change behavior: make sessionReviewIds, sessionApproveAll, projectCwd, vscodeNotOpenWarned imports in the modules that need them. Since they're going to be in src/index.ts, the other modules can't import them (circular dependency would be messy).

    Better approach: move session state to src/review-lifecycle.ts since createReviewAndWait is the primary consumer. Then src/index.ts imports sessionReviewIds and sessionApproveAll from review-lifecycle.ts for the session handlers.
