# Phase 02 -- UI Design Contract

> Visual and interaction contract for Phase 2: Rich Chat Experience.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Platform context:** This phase extends a VS Code extension (`vscode-pi-sr`). There is no web UI, no CSS, no component library, and no design system. All "UI" is rendered through VS Code's native surfaces:
- VS Code Chat panel (progressive markdown via `ChatResponseStream.markdown()`)
- VS Code native dialogs (`window.showQuickPick()`, `window.showInputBox()`, `window.showInformationMessage()`)
- VS Code Settings UI (`package.json` `contributes.configuration`)
- Terminal TUI (unchanged from Phase 1 -- file-based IPC)

The standard UI-SPEC template (spacing scale, typography, color tokens) is designed for web frontends. For this VS Code extension phase, those sections are marked N/A and replaced with a **VS Code Native UI Contract** and **Interaction Contract** that define the actual visual and behavioral surface.

Not applicable. VS Code Chat panel renders markdown using VS Code's UI font stack (controlled by the user's VS Code theme and settings). The extension controls markdown content structure only -- not font family, size, weight, or line height.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | VS Code default | VS Code default | VS Code default |
| Heading | VS Code default | VS Code default | VS Code default |

The extension uses semantic markdown formatting (`**bold**`, ````code blocks````, `---` separators) which adapts to the user's active theme automatically.

## Color

Not applicable. All colors are controlled by the user's active VS Code Color Theme. The extension does not define any custom color values. Semantic markdown rendering (`**bold**`, `` `code` ``, HTML `<details>`) inherits theme colors.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | VS Code theme | Chat panel background |
| Secondary (30%) | VS Code theme | Chat response bubbles, input area |
| Accent (10%) | VS Code theme | Links, selection highlights, command links |
| Destructive | VS Code theme | Error text, warning dialogs |

HTML rendered via `MarkdownString` with `supportHtml = true` is sanitized by VS Code's DOMPurify-based markdown renderer. All styling is theme-inherited; no inline CSS color values are used.

## VS Code Native UI Contract

This section defines the native dialog surface for Pi's `RpcExtensionUIRequest` events (D-11, D-12).

| RPC Method | VS Code API | UI Pattern | Interaction |
|------------|-------------|------------|-------------|
| `select` | `window.showQuickPick(options, { title, placeHolder, ignoreFocusOut: true })` | Quick pick list with single selection | User picks an option or dismisses (Esc) => sends response with `cancelled: true` |
| `confirm` | `window.showInformationMessage(message, { modal: true }, 'Yes', 'No')` | Modal dialog with Yes/No buttons | User clicks Yes => `confirmed: true`; No => `confirmed: false`; Esc => `cancelled: true` |
| `input` | `window.showInputBox({ title, placeHolder, ignoreFocusOut: true })` | Single-line text input box | User types and confirms => `value: input`; Esc => `cancelled: true` |
| `notify` | `window.showInformationMessage()` / `showWarningMessage()` / `showErrorMessage()` | Non-modal notification toast (appears and auto-dismisses) | Fire-and-forget: no response sent back to Pi |
| `editor` (multi-line) | `window.showInputBox({ title, value: prefill, ignoreFocusOut: true })` | Single-line fallback for multi-line editor request | Phase 3 may upgrade to a proper multi-line editor dialog |

Response protocol (sent back via RPC JSON-line):

```typescript
// Success
{ type: 'extension_ui_response', id: string, value: string }
{ type: 'extension_ui_response', id: string, confirmed: boolean }
// Cancellation
{ type: 'extension_ui_response', id: string, cancelled: true }
// Notify: no response
```

Source: RESEARCH.md Code Example 3 (rpc-ui-handler.ts) and Code Example 4 (extension.ts wiring).

## Interaction Contract

Defines the behavioral contract for user interactions with the `@pi` chat participant.

### Streaming Behavior (CHAT-03)

| Property | Specification |
|----------|---------------|
| Pattern | `onEvent()` subscription BEFORE `prompt()` call -- prevents missed events between subscription and message send |
| Event mapping | Per-event: `AgentEvent` -> `mapAgentEventToAction()` -> `applyStreamAction(stream, action)` |
| Tool section buffering | Buffer `tool_execution_start` through `tool_execution_update` events in module-level state; emit complete `<details>` block as single `MarkdownString` on `tool_execution_end` |
| Completion | `waitForIdle()` resolves on `agent_end`; handler returns `{}` |
| Pitfall prevention | Subscribe to `onEvent()` BEFORE calling `prompt()` (Pitfall 1 in RESEARCH.md) |
| Pitfall prevention | Buffer tool events; emit complete HTML block at `tool_execution_end` (Pitfall 3) |

### Interruption Behavior (D-04, D-05, D-06)

| Setting | Value | Default | Behavior |
|---------|-------|---------|----------|
| `pi.chat.interruptionBehavior` | `"abort"` | Yes | User sends new message while streaming: `CancellationToken` fires -> `processManager.abort()` kills current response -> new message starts fresh turn |
| `pi.chat.interruptionBehavior` | `"followUp"` | No | User sends new message while streaming: `CancellationToken` fires -> `processManager.followUp(message)` queues message -> processes after current response completes |

Excluded: `steer()` is not implemented (D-06, deferred to future phase).

### Tool Visibility (D-01, D-02, D-03)

| Setting | Value | Default | Visual Behavior |
|---------|-------|---------|-----------------|
| `pi.chat.toolVisibility` | `"verbose"` | Yes | Full `<details>`/`<summary>` collapsible sections for each tool execution. Progress "Tool: {name}" on start, buffered content emitted on completion. |
| `pi.chat.toolVisibility` | `"quiet"` | No | Single "Pi is working..." progress indicator. No tool names, no partial results, no completion status shown. |

HTML `<details>`/`<summary>` support confirmed: VS Code's DOMPurify-based markdown sanitizer allows these tags (PR #156216). Requires `MarkdownString.supportHtml = true`.

### Slash Command UX (D-07, D-08, D-09, D-10)

| Property | Specification |
|----------|---------------|
| Passthrough | User prompt (including `/command` prefix) sent verbatim to `processManager.prompt(request.prompt)` -- no parsing or interpretation in extension |
| Autocomplete | NOT AVAILABLE: VS Code Chat API 1.82 does not provide `ChatParticipant.slashCommands` or any mechanism for third-party participants to register slash commands for autocomplete. `ChatCommand` type exists in JSDoc but is not a concrete API in `@types/vscode` 1.120.0. |
| Discoverability workaround | Post-response `/help` followup button: `{ prompt: '/help', label: '$(question) Show available commands', kind: vscode.ChatFollowupKind.reply }` |
| Command fetching | `RpcClient.getCommands()` available for internal use (e.g., to populate the followup or for future API expansion). Local RPC over stdin/stdout -- <5ms expected. |

### Terminal TUI Coexistence (CHAT-05, D-13)

| Property | Specification |
|----------|---------------|
| File-based IPC | `.pi/review-requests/` and `.pi/review-results/` protocol is UNCHANGED. No modifications to review request/result JSON schema, file paths, or polling logic. |
| Channel separation | Chat content: VS Code Chat API only. Review requests/results: `.pi/` JSON files only. RPC UI requests: RPC stdin/stdout JSON-line protocol only. Never mix channels. |
| Verification | Terminal `pi` workflow (with TUI selectors) must work without interruption after Phase 2 changes. Existing `.pi/` protocol tests pass. |

## Settings Contract

New VS Code settings introduced in Phase 2 (added to `vscode-ext/package.json` `contributes.configuration`):

| Setting ID | Type | Default | Values | Description |
|------------|------|---------|--------|-------------|
| `pi.chat.toolVisibility` | `string` | `"verbose"` | `"verbose"`, `"quiet"` | Controls tool execution display in chat responses. Verbose shows collapsible sections with details; quiet shows only "Pi is working..." |
| `pi.chat.interruptionBehavior` | `string` | `"abort"` | `"abort"`, `"followUp"` | Controls behavior when user sends a new message while Pi is still streaming. Abort cancels the current response; followUp queues the message for after completion. |

Follows the existing `contributes.configuration` pattern from Phase 1 planning (no Phase 1 settings exist yet; these are the first VS Code settings for the extension).
