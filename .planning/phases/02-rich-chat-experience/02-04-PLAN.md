    **API LIMITATION NOTE (D-08/D-09/D-10 â€” slash command autocomplete):**
    VS Code Chat API 1.82 does not provide a mechanism for third-party chat participants to register slash commands for autocomplete. The `ChatCommand` type referenced in JSDoc is not a concrete API in `@types/vscode` 1.120.0 â€” there is no `ChatParticipant.slashCommands` property or equivalent. Therefore D-08 (autocomplete suggestions when typing `/`), D-09 (fetch commands on each keystroke), and D-10 (all command sources in autocomplete) CANNOT be delivered with the current VS Code API. The fallback approach (Step 4 below) provides a `/help` followup button for command discovery instead. No `getCommands()` call is wired for autocomplete in this plan â€” it remains available on PiProcessManager for future use.

    **Change 2: Create chat handler WITH settings integration**

    Replace this line (line 32):
    ```typescript
    const chatHandler = createChatHandler(processManager);
    ```
    With a wrapper function that reads VS Code settings dynamically:
    ```typescript
    function readChatSettings(): ChatSettings {
      const config = vscode.workspace.getConfiguration('pi.chat');
      return {
        toolVisibility: (config.get<string>('toolVisibility') || 'verbose') as ToolVisibility,
        interruptionBehavior: (config.get<string>('interruptionBehavior') || 'abort') as InterruptionBehavior,
      };
    }

    const chatHandler: vscode.ChatRequestHandler = createChatHandler(processManager, readChatSettings());
    ```

    Note: Per D-09, settings are read once at handler creation time (not per-request). The extension.ts activate function runs once per VS Code session. If a dynamic-per-request read is desired later, the handler can wrap `readChatSettings()` in a closure â€” but for Phase 2, read-once is sufficient since the VS Code Chat API doesn't re-read settings mid-session.

    **Change 3: Wire RPC UI handler in deferred init block (D-12)**

    Inside the existing `void (async () => { ... })()` deferred init block. Add these lines inside, after the heartbeat subscription:

    ```typescript
    // Wire RPC UI handler (D-12): forward extension_ui_request events from Pi to VS Code native dialogs
    const rpcUiHandler = createRpcUiHandler((response) => {
      processManager.sendRpcMessage(response).catch((err: unknown) => {
        console.warn('Pi Companion: failed to send RPC UI response', err);
      });
    });

    processManager.onEvent((event: any) => {
      if (event?.type === 'extension_ui_request') {
        rpcUiHandler(event);
      }
      // AgentEvents are handled by the chat-handler's own onEvent subscription
      // This listener only intercepts extension_ui_request events
    });
    ```

    Key considerations:
    - `processManager.onEvent()` returns an unsubscribe function. Store it so it can be disposed? For Phase 2, the RPC UI handler listener lives for the extension lifetime â€” no need to unsubscribe on cancellation. But store the return in a `context.subscriptions` push if possible.
    - The `event: any` type assertion is required because `onEvent` types its callback as `(event: AgentEvent) => void` but at runtime all JSON lines pass through (including extension_ui_request). Runtime check `event?.type === 'extension_ui_request'` is the defense.
    - Place this code AFTER `reviewCoordinator.start()` and the heartbeat setup, inside the deferred init block's try/catch.

    **Change 4: Add followup provider for slash command discovery (D-08/D-09/D-10 workaround)**

    After the `participant.iconPath` line (line 34), add:
    ```typescript
    // Slash command discovery (D-08/D-09/D-10 workaround):
    // VS Code Chat API 1.82 lacks third-party slash command autocomplete registration.
    // The ChatCommand type is referenced in JSDoc but is not a concrete API in
    // @types/vscode 1.120.0 (no ChatParticipant.slashCommands property).
    // Instead of autocomplete suggestions when typing "/", provide a /help followup
    // button after each response so users can discover available Pi commands.
    participant.followupProvider = {
      provideFollowups(result: vscode.ChatResult, context: vscode.ChatContext, token: vscode.CancellationToken) {
        return [
          {
            prompt: '/help',
            label: '$(question) Show available commands',
            kind: vscode.ChatFollowupKind.reply,
          },
        ];
      },
    };
    ```

    Import `vscode.ChatFollowupKind` is already available from `import * as vscode from 'vscode'`.

    Note: This followup appears after EVERY response. The `kind: reply` means the followup suggestion appears as a reply option (user can click it to send `/help`). Per UI-SPEC Copywriting Contract, the label uses `$(question)` Codicon prefix.
