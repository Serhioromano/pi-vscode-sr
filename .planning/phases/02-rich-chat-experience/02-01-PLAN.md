    Add three methods to the PiProcessManager interface and implement them in the factory's return object.

    Interface additions per D-05, D-08, D-12:
    - `getCommands(): Promise<RpcSlashCommand[]>` — delegates to `state.client.getCommands()`. Import the RpcSlashCommand type by re-declaring a local interface matching the Pi SDK type structure (since the type is not re-exported from the main package entrypoint), or import from `@earendil-works/pi-coding-agent/dist/modes/rpc/rpc-types`.
    - `followUp(message: string): Promise<void>` — delegates to `state.client.followUp(message)`. Signature should accept `message: string` (images parameter not needed for Phase 2 use case).
    - `sendRpcMessage(response: RpcExtensionUIResponse): Promise<void>` — writes a JSON-line extension UI response to the child process stdin. Approach: After `state.client.start()` in the `start()` method, capture a reference to the underlying child process stdin by accessing `(state.client as any).process?.stdin`. Store this in a new `stdin: NodeJS.WritableStream | null` field on the internal state object. In the implementation, `JSON.stringify(response) + '\n'` is written to the stdin stream. Set `state.stdin = null` in `stop()`.
      - The `RpcExtensionUIResponse` type does not need to be imported if you define a minimal local interface: `{ type: "extension_ui_response"; id: string; value?: string; confirmed?: boolean; cancelled?: true }`. This avoids subpath import complications.

    Implementation notes:
    - `getCommands()` throws if `state.client` is null (same pattern as existing `prompt()` method).
    - `followUp()` returns immediately after write; errors should be caught and logged (best-effort by design).
    - `sendRpcMessage()` is fire-and-forget for notify; for select/confirm/input it must block until the stdin write completes (use the callback-style `write()` wrapped in a Promise).
    - Clear `state.stdin = null` in `stop()` alongside `state.client = null`.

    The existing `start()` method already stores `state.client` — add stdin capture right after `await state.client.start()` succeeds.

  <action>
    Create a new module `vscode-ext/src/rpc-ui-handler.ts` that exports a factory `createRpcUiHandler(sendResponse)` returning a function that handles `RpcExtensionUIRequest` events.

    Per D-11 and D-12 and UI-SPEC VS Code Native UI Contract, map each RPC method:

    1. Define local interfaces (imports from subpath SDK types are not guaranteed to be re-exported):
       - `RpcExtensionUIRequest` — discriminated union with `type: string` discriminant, `id: string`, `method: string`, and method-specific fields as defined in rpc-types.d.ts (select: options: string[], confirm: message: string, input: placeholder?: string, editor: prefill?: string, notify: notifyType?: string)
       - `RpcExtensionUIResponse` — discriminated union with `type: "extension_ui_response"`, `id: string`, and result fields (value: string | confirmed: boolean | cancelled: true)
       - `RpcUiHandler` — a function type `(request: RpcExtensionUIRequest) => void | Promise<void>`

    2. Factory `createRpcUiHandler(sendResponse: (response: RpcExtensionUIResponse) => void): RpcUiHandler`:
       - Returns an async function that receives an `RpcExtensionUIRequest`
       - Switches on `request.method`:
         - `select`: `vscode.window.showQuickPick(request.options, { title, placeHolder: "Select an option", ignoreFocusOut: true })`. On selection: `sendResponse({ type: 'extension_ui_response', id, value: selected })`. On dismissal (undefined): `sendResponse({ type: 'extension_ui_response', id, cancelled: true })`.
         - `confirm`: `vscode.window.showInformationMessage(request.title + ': ' + request.message, { modal: true }, 'Yes', 'No')`. Yes: `sendResponse({ type: 'extension_ui_response', id, confirmed: true })`. No: `sendResponse({ type: 'extension_ui_response', id, confirmed: false })`. Esc/dismiss: `sendResponse({ type: 'extension_ui_response', id, cancelled: true })`.
         - `input`: `vscode.window.showInputBox({ title: request.title, placeHolder: request.placeholder, ignoreFocusOut: true })`. On input: `sendResponse({ type: 'extension_ui_response', id, value: input })`. On Esc/dismiss: `sendResponse({ type: 'extension_ui_response', id, cancelled: true })`.
         - `editor`: Same as `input` but using request.prefill as `value` option. Single-line fallback per UI-SPEC (Phase 3 may upgrade to multi-line).
         - `notify`: `vscode.window.showInformationMessage(request.message)` for info/default, `showWarningMessage` for warning, `showErrorMessage` for error. Fire-and-forget — do NOT call sendResponse.
         - `setStatus`, `setWidget`, `setTitle`, `set_editor_text`: Set to no-op with a `console.warn()` (these are not implemented in Phase 2).
         - default: `console.warn('Unknown extension UI request method: ' + request.method)`

    3. Import patterns: use `import * as vscode from 'vscode'` (same as extension.ts).
       Indentation: 2 spaces (vscode-ext convention).
       No default export — use `export function createRpcUiHandler(...)`.

    Key behavioral notes:
    - `ignoreFocusOut: true` prevents dialog dismissal on VS Code focus loss (user must explicitly Esc or select).
    - For `notify`, there is NO sendResponse — Pi does not expect a response for notifications.
    - For `editor` method: single-line `showInputBox` is a Phase 2 limitation per UI-SPEC. Phase 3 may use a multi-line editor.
  </action>
